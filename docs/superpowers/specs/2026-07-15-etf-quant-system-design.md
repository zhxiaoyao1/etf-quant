**日期：** 2026-07-15
**版本：** v2.0
**状态：** 待审核

## 1. 项目概述

### 1.1 项目目标

构建一个纯手机端运行的双引擎量化分析 PWA 应用：

- **场内 ETF 引擎**：基于技术指标，每日生成择时信号（买/观望/卖）
- **场外基金引擎**：基于基本面多维度评分，每周生成选基信号 + 持仓诊断信号

两个引擎各自拥有独立的因子体系和自学习机制，共享统一的三色信号（🟢买入 ≥80 / 🟡观望 40-79 / 🔴卖出 <40）。

### 1.2 核心特性

- 📱 **纯手机运行**：PWA 应用，所有计算和存储均在手机浏览器中完成
- 📊 **双引擎量化**：场内 ETF（技术面 4 因子）+ 场外基金（基本面 5 因子）
- 🧠 **双轨自学习**：各自独立跟踪信号准确率，自动调整因子权重
- 🟢 **统一信号体系**：所有场景均使用 ≥80 买入 / 40-79 观望 / <40 卖出
- 🔧 **高度可配置**：所有参数可在设置页面调整，无需改代码
- 🧩 **因子可扩展**：新增因子只需新建一个文件并注册
- 📴 **离线可用**：Service Worker 缓存，无网络也能查看历史数据

### 1.3 用户画像

基金投资入门用户，同时关注场内 ETF 和场外基金，了解基本概念（均线、RSI、夏普比率等），希望有一个统一的辅助决策工具，通过系统自学习能力逐步提升策略有效性。

### 2.2 关键技术决策

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 运行平台 | 纯 PWA（手机浏览器） | 一套代码 iOS+Android，无需应用商店，即时更新 |
| 计算位置 | Web Worker 后台线程 | 量化计算不阻塞 UI，保持页面流畅 |
| 数据存储 | IndexedDB | 浏览器端持久化存储，容量大，支持复杂查询 |
| ETF 数据来源 | AKShare / 东方财富免费接口 | 零成本，盘后分析足够 |
| 场外基金数据 | 天天基金 / 好买基金 / AKShare 基金模块 | 净值、排名、经理、季报数据 |
| 前端框架 | React + TypeScript | 生态成熟，类型安全，组件化开发 |
| 构建工具 | Vite + vite-plugin-pwa | 快速构建，内置 PWA 支持 |
| 图表库 | Lightweight Charts | 轻量高性能，适合移动端 K 线图 |

### 2.3 部署方式

1. 构建产物为静态文件（HTML + JS + CSS）
2. 部署到任意静态托管服务（GitHub Pages / Vercel / Cloudflare Pages）
3. 手机浏览器打开 URL → 添加到主屏幕 → 体验等同原生 App
4. Service Worker 自动缓存，后续打开秒开

## 3. 量化模型设计

### 3.1 场内 ETF 引擎（技术面，每日）

#### 3.1.1 四因子体系

每个因子独立输出 0-100 分，分数越高越看多。

**趋势因子（Trend）**
- 指标：MA(5), MA(20), MA(60), MACD(12,26,9)
- 均线多头排列 → 80-100 分；空头排列 → 0-30 分；缠绕 → 40-60 分

**动量因子（Momentum）**
- 指标：RSI(14), KDJ(9,3,3)
- RSI 30-50 区间 → 高分（超卖反弹机会）；RSI > 70 → 低分（超买风险）

**波动率因子（Volatility）**
- 指标：布林带(20,2), ATR(14)
- 接近下轨 → 高分（低位）；接近上轨 → 低分（高位）；ATR 异常放大 → 降低分数

**资金流因子（Money Flow）**
- 指标：成交量变化率, OBV 趋势, 量比
- 放量上涨 → 高分；放量下跌 → 低分；OBV 背离 → 修正

#### 3.1.2 综合评分公式

```
ETF 综合评分 = S_trend × W_trend + S_momentum × W_momentum
             + S_volatility × W_volatility + S_moneyflow × W_moneyflow

约束：W_i ∈ [0.10, 0.50]，ΣW_i = 1.0，S_i ∈ [0, 100]
W_i 由自学习模块每周更新
```

### 3.2 场外基金引擎（基本面，每周）

#### 3.2.1 选基五因子（基金筛选）

| 因子 | 核心指标 | 权重范围 |
|------|----------|----------|
| 收益能力 | 近1/3/6月·1/3年收益, 超额收益(vs基准) | 10%-50% |
| 风险控制 | 最大回撤, 年化波动率, 下行标准差 | 10%-50% |
| 风险调整收益 | 夏普比率, 索提诺比率, 卡玛比率, 信息比率 | 10%-50% |
| 经理质量 | 任职年限, 管理规模, 任职回报 | 10%-50% |
| 基金基本面 | 规模, 费率, 成立年限, 持有人结构 | 10%-50% |

```
选基综合评分 = S_return × W_return + S_risk × W_risk + S_riskadj × W_riskadj
             + S_manager × W_manager + S_fundamental × W_fundamental
```

#### 3.2.2 持仓诊断五维度（已持有基金体检）

| 维度 | 检查项 | 触发预警条件 |
|------|--------|-------------|
| 相对表现 | vs 同类平均, vs 基准指数 | 连续3个月跑输同类后50% |
| 风险异动 | 当前回撤, 波动率突变 | 回撤超过历史最大回撤 80% |
| 经理变动 | 是否换人, 是否一拖多 | 经理变更 / 同时管理 >5只 |
| 风格漂移 | 基金风格 vs 名义风格 | 持仓偏离名义风格 >30% |
| 规模健康 | 规模变化, 大额申赎 | 单季度规模翻倍/腰斩 |

```
诊断健康度 = S_performance × W_perf + S_risk × W_risk + S_manager × W_mgr
           + S_style × W_style + S_scale × W_scale
```

### 3.3 统一信号体系

**所有场景均使用以下阈值：**

| 综合评分 | 信号 | ETF（每日） | 场外-筛选（每周） | 场外-诊断（每两周） |
|----------|------|------------|------------------|-------------------|
| ≥ 80 | 🟢 买入 | 买入 | 可买入 | 继续持有 |
| 40-79 | 🟡 观望 | 观望 | 可关注 | 减仓观察 |
| < 40 | 🔴 卖出 | 卖出 | 不推荐 | 建议换基 |

> **设计意图：** ≥80 的高门槛意味着多因子高度共振才触发买入信号，减少噪音和频繁交易。40 分的宽观望区间避免今买明卖。阈值可在设置中手动调整，默认值 80。

## 4. 自学习机制（双轨）

### 4.1 学习循环

```
发信号 → 等待验证 → 评估各因子准确率 → 调整权重 → 下次用新权重
```

三个场景各自独立学习：

| 场景 | 频率 | 学习目标 | 回看窗口 |
|------|------|----------|----------|
| ETF 择时 | 每周 | 优化 4 因子权重，提高择时准确率 | 最近 20 条信号 |
| 场外选基 | 每月 | 优化 5 因子权重，提高选基命中率 | 最近 1 年数据 |
| 场外诊断 | 每月 | 优化诊断触发阈值，减少误报 | 最近 10 次诊断 |

### 4.2 权重调整算法

**ETF 引擎**（以 ETF 为例，场外同理）：

**输入：** 最近 N 条信号记录，每条包含各因子得分 + N 天后实际涨跌

**步骤：**

1. **计算各因子近期准确率**
   - 因子得分 > 50 且实际涨 → 预测正确
   - 因子得分 < 50 且实际跌 → 预测正确
   - 准确率 = 预测正确次数 / 总信号数

2. **按准确率重分配权重**
   ```
   新权重_i = 准确率_i / Σ(准确率_j)
   ```

3. **平滑处理**
   ```
   最终权重_i = 旧权重_i × (1 - α) + 新权重_i × α
   ```
   学习率 α 默认 0.3，可配置。

4. **边界裁剪**
   - 权重 < 10% → 裁剪为 10%
   - 权重 > 50% → 裁剪为 50%
   - 重新归一化

**场外选基引擎**：逻辑相同，验证标准为「筛选时评分 ≥80 的基金在下一季度是否跑赢同类平均」。

**场外诊断引擎**：验证标准为「诊断时建议换基（<40）的基金在接下来 3 个月是否确实持续走弱」。

### 4.3 安全机制

| 机制 | 说明 |
|------|------|
| 最小样本量 | ETF 不足 10 条 / 场外不足 5 次时不触发学习 |
| 权重上下限 | 每因子 10%-50%，避免单一因子权重过高/过低 |
| 平滑系数 | 默认 0.3，让权重缓慢变化 |
| 人工锁定 | 用户可在设置中锁定某个因子的权重 |
| 透明度 | 每次调整记录日志：时间、各因子旧权重→新权重、调整原因 |

## 6. 可扩展性设计

### 6.1 配置驱动

所有可调参数集中在 `config/defaults.ts`，包括：
- 各因子指标参数（均线周期、RSI 周期、布林带参数等）
- 信号阈值（默认买入 80 / 卖出 40）
- 自学习参数（学习率、回看窗口、权重上下限）
- 数据源配置

设置页面读取配置自动生成 UI，修改后实时生效。

### 6.2 因子插件化

每个因子是一个独立模块，遵循统一接口：

```typescript
interface Factor {
  id: string;
  name: string;
  description: string;
  type: 'etf' | 'mutual-fund-screening' | 'mutual-fund-diagnosis';
  calculate(data: KLine[] | FundData): number;  // 输出 0-100 评分
  params: Record<string, number>;
}
```

新增因子只需：
1. 在对应 `factors/` 子目录下新建文件，实现 `Factor` 接口
2. 在注册表中添加一行
3. 设置页面自动出现新因子的配置项

### 6.3 数据层解耦

```
数据获取(Fetcher) → 数据清洗(Cleaner) → 数据存储(Store) → 分析引擎消费
```

- ETF Fetcher: AKShare + 东方财富
- 场外 Fetcher: 天天基金 + 好买基金 + AKShare 基金模块
- 换数据源只需替换 Fetcher，不影响下游

## 7. 数据流

### 7.1 ETF 每日分析流程

```
1. 用户打开 App / Service Worker 后台触发
2. etfFetcher 从 AKShare 拉取 ETF 日线数据
3. Cleaner 清洗 → Store 存入 IndexedDB
4. 主线程 → postMessage → ETF Worker
5. Worker 中：四因子评分 → 加权综合 → 生成信号 → 信号入 IndexedDB
6. Worker → postMessage → 主线程 → UI 更新看板
```

### 7.2 场外基金每周分析流程

```
1. fundFetcher 从天天基金等拉取净值、排名、经理数据
2. Cleaner 清洗 → Store 存入 IndexedDB
3. 主线程 → postMessage → Fund Worker
4. Worker 中：
   a. 选基：五因子评分 → 加权综合 → 排序 → TOP 基金列表
   b. 诊断：五维度检查 → 健康度评分 → 信号 + 预警明细
   c. 结果存入 IndexedDB
5. Worker → postMessage → 主线程 → UI 更新选基页
```

### 7.3 自学习流程

```
ETF（每周）：
  读取最近 20 条信号 → 验证涨跌 → 计算各因子准确率 → 调整权重 → 记录日志

场外选基（每月）：
  读取上月筛选结果 → 验证本季表现 → 调整选基因子权重 → 记录日志

场外诊断（每月）：
  读取历史诊断 → 验证后续走势 → 调整诊断阈值 → 记录日志
```

## 9. 风险与约束

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| AKShare 接口不稳定 | 数据更新失败 | 多数据源备用 + 手动导入入口 |
| 天天基金接口限制 | 场外数据获取失败 | 缓存 + 降频 + 多源切换 |
| 手机存储空间 | IndexedDB 满了 | 设置数据保留期限，自动清理 |
| iOS PWA 限制 | 后台同步不可靠 | 用户打开 App 时主动触发更新 |
| 浏览器兼容性 | 部分 API 不可用 | 降级方案（无 Worker 时主线程计算） |
| 场外基金评价滞后 | 季报数据延迟 | 以净值数据为主，季报数据为辅 |
